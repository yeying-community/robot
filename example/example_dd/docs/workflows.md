# dd-bot 处理流程设计

## 1. 总体消息主链路

```text
钉钉群消息
  -> 消息标准化
  -> 上下文补齐
  -> 意图识别
  -> 路由到对应工作流
  -> 生成回复 / 草稿 / 外部动作预览
  -> 确认后写入外部系统
  -> 结果回写群聊
```

## 2. 消息标准化

收到原始消息后，需要先统一处理：

- 去掉钉钉格式噪声
- 识别 `@机器人`
- 拆分 URL、代码片段、错误码
- 统一文本空格和换行
- 记录附件与链接
- 补充策略配置、仓库映射、Wiki 空间建议

## 3. 上下文补齐

每条消息处理前，补充以下上下文：

- 最近 N 条群消息
- 当前讨论线程候选
- 发言人身份
- 发言人是否新人
- 群配置
- 运行时策略配置
- 是否存在未完成草稿

## 4. 意图识别

### 4.1 分类结果

- `ignore`
- `qa`
- `summary`
- `calendar`
- `issue`
- `wiki`
- `mixed`
- `handoff`

### 4.2 推荐策略

- 规则优先
- 模型辅助
- 问答前先本地检索
- 高风险动作必须再过策略层

## 5. 问答工作流

### 5.1 进入条件

- 消息像一个问题
- 问题在知识库覆盖范围内
- 触发策略允许自动答

### 5.2 处理步骤

1. 识别问题主题
2. 选择知识过滤条件
3. 用本地检索工具执行检索
4. 评估命中质量
5. 决定直接答 / 澄清 / 转人工
6. 如知识不足，则记录待补知识项
7. 输出带来源的回复或统一兜底回复

### 5.3 决策规则

- 高置信度：直接答
- 中置信度：先问一个关键问题
- 低置信度：不乱答，转人工或引导文档

## 6. 总结工作流

### 6.1 进入条件

- 用户显式要求总结
- 或系统命中自动整理策略

### 6.2 处理步骤

1. 选择消息时间范围
2. 聚合相关消息
3. 提炼主题、背景、结论、待办、风险
4. 输出群摘要
5. 可选生成 Wiki / Issue 候选

## 7. 日程工作流

### 7.1 进入条件

- 消息表达出明确会议意图

### 7.2 处理步骤

1. 抽取日期、时间、主题、参会人
2. 缺字段则追问
3. 生成日程预览
4. 保存待执行动作
5. 由发起人或管理员确认
6. 调用钉钉日程接口
7. 回写创建结果

### 7.3 关键要求

- 不完整信息不直接创建
- 创建前必须确认
- 创建失败后保留预览数据

## 8. Issue 工作流

### 8.1 进入条件

- 用户显式提出“提 Issue”
- 或机器人识别出需要跟踪的 Bug / 任务

### 8.2 处理步骤

1. 提炼问题背景
2. 生成标题、仓库和描述
3. 推断类型、优先级、负责人建议
4. 生成 GitHub Issue 草稿
5. 保存待执行动作
6. 由发起人或管理员确认
7. 提交到 GitHub
8. 回写 Issue 链接

### 8.3 关键要求

- 项目仓库不明确时先追问
- 不自动指定不确定的负责人
- 失败时保留草稿

## 9. Wiki 工作流

### 9.1 进入条件

- 用户要求沉淀
- 或总结结果被标记为适合转 Wiki

### 9.2 处理步骤

1. 聚合消息和背景知识
2. 选择 Wiki 模板
3. 生成 Markdown 草稿
4. 请求确认
5. 写入公司 Wiki（当前仍未接入）
6. 将审核通过版本标记为可入知识库候选

## 10. 混合意图处理

典型场景：

- “总结一下并提个 issue”
- “整理成 wiki，顺便约个会”

处理原则：

- 先拆成多个子任务
- 先做只读和草稿
- 再做写操作确认

## 11. 失败与兜底

### 11.1 问答失败

- 明确说未找到可靠答案
- 引导查看文档或找负责人

### 11.2 外部系统失败

- 提示写入失败
- 保留草稿
- 不丢失上下文

### 11.3 高风险场景

以下情况直接转人工：

- 线上事故
- 生产权限
- 敏感数据
- 不明确责任归属

# dd-bot 产品需求文档

## 1. 产品目标

在一个钉钉研发群里，构建一个可控、可追溯的协作机器人，使其能够：

- 对知识类问题给出基于知识库的回答
- 自动整理群聊中的结论、待办和风险
- 为日程、Issue、Wiki 生成可审阅的草稿
- 在确认后调用外部系统执行写操作

## 2. 用户与角色

### 2.1 用户角色

- **新人**
- **普通研发成员**
- **群负责人 / 导师**
- **机器人维护者**

### 2.2 角色差异

- 新人问题更适合自动答复
- 老同学问题默认更偏辅助而非抢答
- 管理员拥有更高的写操作确认权限

## 3. 功能范围

### 3.1 知识问答

机器人需要支持：

- 新人 onboarding 问答
- 流程问答
- 文档导航
- FAQ 类问题

回答要求：

- 回答短、结构化
- 基于知识库
- 带来源
- 先检索再回答
- 低置信度时澄清或转人工

### 3.2 群聊整理

机器人需要支持：

- 生成讨论摘要
- 提炼关键结论
- 提取行动项
- 识别风险和阻塞
- 标记 FAQ 候选

### 3.3 日程辅助

机器人需要支持：

- 识别会议 / 同步意图
- 提取时间、主题、参与人
- 缺字段时追问
- 生成钉钉日程预览
- 确认后创建日程

### 3.4 Issue 辅助

机器人需要支持：

- 识别需要跟踪的问题
- 生成 GitHub Issue 草稿
- 提炼标题、目标仓库、描述、优先级建议、来源消息
- 确认后提交到 GitHub

### 3.5 Wiki 辅助

机器人需要支持：

- 识别需要沉淀的群讨论
- 生成公司 Wiki 草稿
- 给出空间 / 分类建议
- 按统一模板输出结构化内容
- 审核后发布

## 4. 触发规则

### 4.1 自动触发

允许自动触发的场景：

- 新人的知识问答
- 明确的“帮我总结一下”
- 明确的“整理成纪要 / Wiki / Issue 草稿”

### 4.2 半自动触发

需要机器人先识别，再二次确认的场景：

- 创建钉钉日程
- 提交 GitHub Issue
- 发布 Wiki

### 4.3 不触发场景

默认不响应：

- 表情
- “收到”“ok”“我看下”
- 明显闲聊
- 明显高风险敏感话题

## 5. 意图分类

第一版至少要支持以下意图：

- `ignore`
- `qa`
- `summary`
- `calendar`
- `issue`
- `wiki`
- `mixed`
- `handoff`

## 6. 输出规范

### 6.1 问答输出

统一模板：

- 结论
- 步骤
- 注意事项
- 来源文档
- 不确定时找谁

### 6.2 总结输出

统一模板：

- 主题
- 背景
- 结论
- 行动项
- 风险
- 相关人

### 6.3 草稿输出

草稿类输出必须包含：

- 草稿标题
- 草稿类型
- 关键信息字段
- 需要确认的字段
- 来源消息范围
- Issue 草稿必须包含目标仓库
- Wiki 草稿必须包含空间 / 分类建议

## 7. 权限与确认

### 7.1 动作分级

- **只读**：问答、摘要、检索
- **草稿**：Issue 草稿、Wiki 草稿、日程预览
- **写入**：创建日程、提交 Issue、发布 Wiki

### 7.2 默认策略

- 只读可自动执行
- 草稿可自动生成
- 写入必须确认
- 写入默认只允许发起人或管理员确认 / 取消

## 8. 非功能要求

### 8.1 可追溯

- 必须记录消息来源
- 必须记录意图判断结果
- 必须记录知识命中结果
- 必须记录外部动作执行结果

### 8.2 可控

- 不能抢答所有消息
- 不能在低置信度时胡乱下结论
- 不能在无确认时写外部系统

### 8.3 可维护

- 规则、配置、连接器都要可替换
- 文档与代码目录清晰
- 需要为后续接 `yeying-rag` 留接口

## 9. MVP 验收标准

以下能力同时满足，才算 MVP 完成：

- 能接收指定研发群消息
- 能稳定回复知识库问答
- 能输出群讨论摘要
- 能生成 GitHub Issue 草稿
- 能生成 Wiki 草稿
- 能给出钉钉日程预览
- 写操作需要确认
- 全链路有日志

## 10. 关键边界

第一版不要做：

- 全群自动陪聊
- 复杂长期记忆
- 自动把未审核群聊写回正式知识库
- 自主执行高风险决策
